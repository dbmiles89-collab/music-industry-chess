## 18 Jul — Scaffolding React + Laravel
PHP is a back-end language for servers. Learning about the DOM in front-end web development, I understand that programs such as Lit and React were created to easily create and update DOM templates. These programs make it easier to translate what is wanted into plain JavaScript, doing the heavy lifting for the developer. This is similar to the relationship between PHP and Laravel on the back end. It is the same concept ontologically, even if it is different at a substrate level. Realizing this helped me understand frameworks as abstractions over a language's pain points across both tiers of the stack.
I used npm and Composer in the command line to fetch the scaffolding skeletons and download the initial libraries and commands I will need. Specifically, npm create vite and composer create-project fetched these skeletons.
Running npm run dev and php artisan serve launched the development servers.
This created two separate servers. I initially was confused as to why the front end and back end need separate servers, but I now understand that this is a scaffolding process for the development stage, not the app's final form. Front-end code does not actually need a server to execute. A server can simply hold JavaScript files react compiles as data and hand them over to a browser, which is what actually speaks and understands JavaScript.  However, in the beginning, it is much better to build them separately for clarity.
## 18 Jul – Understanding the Architecture
The next move was to rebuild my chessboard. In my original code, where I hand-rolled a lot of the DOM elements as well as the chess logic (genMoves and drawPieces), I realized that my prototype conflated two concerns: game rules and DOM rendering. I am now rebuilding it in React, with chess.js owning the rules and React owning the rendering, which was a great joy to learn! It simplifies my understanding, as each encapsulation and compartmentalisation of the code helps me zoom out and in to understand the details and the broader scope of what I am doing.
## 19 July Imports, Components and Structure
App failed silently, I managed to trac it to chess.js never being installed. Lesson: an import needs its package installed in package.json, not just written in code. I'm starting to understand the building blocks React encourages you to think in. I defined the board in Board.jsx (inside client/src), and learned about useState as a hook — React's way of remembering the chess.js brain between re-renders rather than recreating it. Then in App.jsx I used <Board /> as a tag, which taught me React's flow at the substrate level: a component is defined as a function; a tag invokes that function; the function returns a description of the UI (the virtual DOM); and then under the hood React turns that description into the actual DOM. I also clarified a language fundamental: function in JavaScript is like def in Python — it defines behaviour. Thinking in building blocks on the component tree lets me zoom out to the high-level structure, then zoom back into the detail of what each component does.
